# ADR-0005. Выделить цифровой двойник в отдельный сервис

## Статус

Принято.

## Контекст

Цифровой двойник в проекте должен накапливать сигнатуры, историю событий, результаты осмотров, состояние участков и тренды. Его развитие может идти быстрее и глубже, чем развитие базового pipeline инцидентов: после MVP могут появиться прогноз деградации, рекомендации обслуживания, учет метеоданных и расширенные модели состояния.

## Решение

Считать цифровой двойник отдельным сервисом, который:

- получает подтвержденные события и результаты осмотров;
- читает временные ряды и агрегаты признаков;
- обновляет индекс состояния участка;
- хранит историю изменений состояния;
- отдает API для АРМ и аналитики.

В MVP цифровой двойник не является источником автоматических эксплуатационных команд.

## Альтернативы

| Альтернатива | Почему не выбрана |
|---|---|
| Встроить цифровой двойник в Backend API | Быстрее в первой версии, но смешивает карточки инцидентов и аналитическую модель состояния |
| Встроить в ML workers | Неправильная ответственность: ML классифицирует события, а цифровой двойник ведет состояние участка |
| Отложить цифровой двойник полностью | Противоречит концепции проекта и снижает ценность накопленных данных |

## Последствия

Плюсы:

- четкая граница ответственности;
- проще развивать прогноз и рекомендации;
- можно отдельно тестировать обновление состояния участков;
- API и ML pipeline не перегружаются логикой долгосрочного состояния.

Минусы:

- появляется дополнительный сервис;
- нужны правила согласованности между инцидентами, заданиями и состоянием участка;
- в MVP часть функций цифрового двойника будет базовой.

## Когда пересмотреть

Решение стоит пересмотреть, если после MVP цифровой двойник останется только статическим полем в карточке участка и не потребует отдельной логики, данных и развития.
