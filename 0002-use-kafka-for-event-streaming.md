# ADR-0002. Использовать Kafka для потоковой обработки событий

## Статус

Принято.

## Контекст

Система получает события от множества edge-узлов, классифицирует их, оценивает критичность, запрашивает видео, создает инциденты и обновляет цифровой двойник. Эти этапы имеют разную скорость обработки и должны масштабироваться независимо. Повторная доставка событий возможна и должна быть безопасной.

## Решение

Использовать Kafka как основной брокер потоковых событий между edge-узлами, ML workers, сервисом критичности, Backend API и цифровым двойником.

Минимальный набор topics:

- `edge.events`;
- `ml.classifications`;
- `incidents.commands`;
- `incidents.events`;
- `twin.updates`.

## Альтернативы

| Альтернатива | Почему не выбрана |
|---|---|
| RabbitMQ | Хорош для очередей задач, но хуже подходит для потоков и повторного чтения событий аналитикой |
| HTTP-вызовы между сервисами | Жестко связывают компоненты и плохо переносят всплески событий |
| PostgreSQL как очередь | Упрощает инфраструктуру, но создает нагрузку на источник истины и хуже масштабируется |

## Последствия

Плюсы:

- независимое масштабирование consumers;
- backpressure через lag;
- возможность подключать новые обработчики событий;
- естественная интеграция с наблюдаемостью pipeline.

Минусы:

- повышается сложность эксплуатации;
- нужны правила схем событий и совместимости;
- требуется аккуратная идемпотентность consumers.

## Когда пересмотреть

Решение стоит пересмотреть, если поток событий окажется небольшим, команда не готова сопровождать Kafka, а простая очередь задач покроет все сценарии MVP.
